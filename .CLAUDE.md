
- This codebase is intended to be open-source and collaborative. Please follow these guidelines to ensure that your contributions are high-quality and maintainable.

- Minimize COSMETIC CHANGES (e.g., changing a parameter name of a method or a comment unnecessarily)
- Before making any change, understand the full context by reading as much code as necessary. Do not guess what you can know for certain by reading code.

- No hacky workarounds. No slop or code bloat.
- Do not just focus on fixing the symptoms of the problem. Fix the root cause.
- DO NOT OVERCOMPLICATE THE CODE.

- List all the issues in the log messages and work on the root cause of each one. Reduce the guesswork. Find all instances of each issue in the code.

- Fail the code if it violates expectations. Data needs to be typed correctly. Objects need to have the expected attributes and methods. If something is not as expected, raise an error immediately with a clear message. Do not let bad data propagate through the system.

- Identify the root cause of an issue and go for the radical solution that eliminates the problem once and for all - no band-aid.

- Follow best practices for code quality, readability, and maintainability. Write clean, well-structured code with clear variable names and comments where necessary. Avoid unnecessary complexity and keep functions and methods focused on a single responsibility.

- Unless explicitly requested, only make surgical changes to the codebase to address the specific issue at hand. And avoid making cosmetic or unncessary changes (e.g., changing parameter names, comments, or formatting) that do not directly contribute to fixing the problem. This helps maintain a clear and concise commit history and help with code review.

- Always save the findings of extensive code exploration in local files and keep track of them. This will help to preserve important insights and context that may be useful for future reference and avoid losing valuable information or progress status due to context compaction.

- Always be focused and surgical (no cosmetic or unnecessary or unrelated changes) and thorough (keep track of your progress against the plan), and plan for context compaction (do not wait until your context is at 100%; proactively save progress and findings in local files you can reference later even context is compacted).

- Always keep track of your progress in a todo list. Be careful and paranoid about making any changes in the codebase. Read as much code as necessary to ensure you are not breaking something else while fixing a bug. Be honest and proactive. No lazy or hacky or sloppy code. This codebase requires the highest coding standards.

- When reading logs, list all found errors and start debugging them one by one and keep track of your progress in a todo list. No hacky workarounds or sloppy bandaid. Only genuine, disciplined fixes that are architecturally sound

- Bug fixes should not only cover the issues reported (which would be bandaid) but also should behave correctly in all other situations. So first try to articulate what the correct behavior is in all cases, then ensure there are runtime checks in the right places to catch any misuse by client code and ensure that the correct behavior is followed. No cosmetic or unnecessary changes. You can add more trace points to narrow down the issue with a few more test runs.


- Think about tasks given this constraint: minimal throwaway work on my side (review, testing, debugging and bug fixing). Vast majority of the code you write needs to be durable and stand the test of time (given the long-term vision). I am not saying "completely avoid throwaway work". But strike a balance between moving fast to a working prototype (to resolve all unknowns and uncertainties) and minimizing throwaway work.

- When I push back, your default should be "re-derive from first principles, don't refine the prior answer."

- Always update the `colony/docs` with all the new capabilities because these aspects are crucial for the adoption of this open-source library. Docs need to be concrete, concise and expository. Explain concepts and provide examples. Do not be vague or obtuse.

- All changes in code, docs or the proposal need to be surgical and minimal to make it easier to review code changes. No cosmetic or unnecessary changes (e.g., removing comments or whitespace or altering docstring unnecessarily).

- Read any code before you edit it. No blind rewrites.

- Do not make the mistake of underestimating the amount of work in a PR during the planning phase because you got too lazy to read the codebase. Do your due diligence.

- Do not leave any technical debt any PR.

- Document PRs so that operators and users know what they are expected to do (env vars, app creation on GitHub, etc.) and do not leave them to guess.

- Do not only fix the bug I point out without considering the possibility that it is more systematic than the particular instance I pointed out. Do not treat each call-out as a single instance instead of an **audit prompt**.

- Whenever you make a code change, investigate its implications and ripple effects not just immediate effects. When you add a helper / pattern / fix, grep all call sites with the same input shape or that do the same kind of thing. Update them in the same PR or call out follow-ups explicitly. Default-silent is the failure mode.

- Before creating a new helper / context manager / decorator / utility, **search the surrounding scope first** for an existing one that does the same thing or an adjacent thing. Grep the file's directory for one with the same shape (not just the same name). If something close exists, extend it rather than parallel-implement. If you do create a new one, in the docstring, name the related helpers in the same directory and explain why a new one is justified. This forces you to do the search and prevents future-me from re-duplicating.


- Do not introduce unnecessary couplings across the codebase (e.g., using a property for its side effect (triggering the lazy clone) with the intent hidden behind an unrelated name, or  pulling a private constant across the encapsulation boundary) Add methods with explicit intent instead.


- **Verify wiring claims by reading the called code, not the surrounding doc**. If a README says "automatically built by X," open X and grep for the build trigger. If a section says "the planner reads Y via Z," open Z and verify it has the read surface for Y.

- **For every new primitive (global, lock, cache, file path), enumerate the four contexts**: sync single-thread, async single-task, async concurrent-tasks, nested-callers. Pick the primitive that survives all four. Default to ContextVar for cached session-like state, not a module global.

- **For every artifact you drop into a directory, answer: "who reads this at runtime?" before deciding the location**. If the answer is "the agent," it's package data. If "humans only," it's docs. If "both," it's package data + a docs-side pointer.

- First identify (and agree on) the general requirements for any feature (e.g., trial procedures need to be pure or at least have consistent behavior when invoked as actions on an agent or invoked as standalone trial runnables), then audit the existing codebase for any incompatibilities with those requirements, then find solutions for those incompatibilities (e.g., isolate them in new or extended data types, etc.) and finally conduct audits of every requirement in the final implementation.

- If you can't solve a problem, just admit it and declare the problems (e.g., conflicting requirements) that made you stuck instead of forcing me to waste time reviewing broken code.


- Read the whole file end-to-end before the first edit on it
- Surgical edits only — no cosmetic rewrites, no reformatting, no touching code unrelated to the task
- No fabrications — verify before claiming; read the actual code, don't infer from naming or memory
- Audit the class, not just the instance — when changing a behavior, check every call site and every sibling that depends on the same convention
- Reduce LLM-planner cognitive load — kwargs the planner can hallucinate get derived from capability state instead of accepted as arguments
- Multi-tenant correctness — per-tenant state belongs in tenant rows, per-user in user rows, never in deploy-wide env or guessed from convention
- No blind rewrites — prefer surgical Edit over Write on existing files
- Pre-edit gates: cross-repo / private-import / read-before-edit / cross-object private-attribute
